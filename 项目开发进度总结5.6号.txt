## 一、 项目核心目标
利用Python及开源工具链，构建一个运行在本地Windows系统、零封号风险的微信消息管理工具。
核心价值：将个人账号中的海量、分散的聊天信息进行本地聚合、智能过滤和待办提醒，解决“信息过载”和“遗忘”问题。

## 二、 技术架构与核心设计
### 1. 数据层：
- 利用 `wechat-cli` 命令行工具作为核心接口，负责对微信本地加密数据库进行**只读解密**和结构化JSON输出。
- 自建本地SQLite数据库 (`cache.db`)，用于持久化存储所有拉取的聊天记录、联系人、设置项和同步进度。
- **快照机制**：软件关闭时，将界面数据 (`all_messages_list`) 序列化为 `snapshot.json` 文件。再次启动时，直接加载快照实现界面“秒开”，后台静默更新增量数据，模仿聊天软件的启动体验。

### 2. 核心引擎 (Engine)：
- 采用 **“生产者-消费者”** 与 **“小组长+工人”** 混合架构，将繁重任务分解到独立线程中，避免阻塞GUI主线程。
- **小组长 (MessageEngine)**：负责总调度。启动时根据`sync_progress`表决定策略（首次全量/后续增量），并管理日常的轻量级轮询。
- **工人1：拉取 (SyncWorker / IncrementalSyncWorker)**：运行在独立`QThread`中。首次启动用`ThreadPoolExecutor`（10个工人）并行拉取历史消息；后续启动仅拉取增量。拉取进度直连界面。
- **工人2：分析 (Analyzer，规划中)**：未来将独立负责AI文本分析、任务提取。
- **工人3：语音 (VoiceWorker，规划中)**：未来将独立负责语音转文字。

### 3. 界面层 (GUI)：
- **技术栈**：PyQt5，采用系统托盘常驻运行模式。
- **核心页面**：
  - **实时消息**：仿微信会话列表，按最新消息时间排序，支持黑名单过滤、工作/紧急关键词高亮。
  - **设置窗口**：独立弹出窗口，分6大分类（消息监控、信息过滤、提醒方式、数据管理等），支持关键词/黑名单增删改查，并持久化到数据库。
  - **微光进度条**：自定义`ShimmerProgressBar`控件，提供一道淡紫/淡金色的细长光斑流动动画，在同步时显示。
  - **紧急提醒弹窗**：右下角置顶弹窗，用于紧急关键词通知。

### 4. 数据流：
- **启动流程**：加载`snapshot.json`(秒开) -> 界面渲染 -> 后台启动增量/全量同步工人。
- **同步流程**：`SyncWorker`(并行/串行) -> 拉取`wechat-cli`数据 -> 写入本地DB -> 更新同步进度 -> 发射进度/完成信号 -> 界面更新。
- **日常轮询**：`get_new_messages()` -> 过滤黑名单/打标签 -> 写入DB -> 节流刷新界面(每2秒) -> 紧急弹窗。

## 四、 主要性能优化与Bug修复历程
1.  **数据库写入优化**：从逐条`commit`改为`executemany`+大事务(1000x性能提升)。
2.  **UI渲染优化**：引入节流刷新（每2秒），彻底解决拖动窗口卡顿。
3.  **拉取架构优化**：`SyncWorker`从单线程串行改为`ThreadPoolExecutor`并行(10 workers)，拉取速度提升5-10倍。
4.  **时间解析Bug修复**：解决微信返回的纯时间格式（如"10:21"）导致排序混乱和`UnboundLocalError`。
5.  **信号链路持久战**：进度信号 (`progress_signal`) 百分比不显示/进度条不消失的终极原因，被锁定在多线程环境下的`QTimer`延迟连接与信号跨线程转发的竞争和丢包问题。最终解决方案是**信号直连**（Worker -> MainWindow），砍掉所有中间转发环节。
6.  **数据缺失修复**：解决因时间戳格式不一致，导致`get_incremental_messages`未能获取全部新消息的问题。
